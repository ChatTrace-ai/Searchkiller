Searchkiller Cloud Run 部署流程

这是一份面向 Searchkiller 的 Cloud Run 部署 SOP，覆盖镜像构建、服务部署、Secret 配置、权限授权、 验证方式和常见构建问题。命令中的 PROJECT_ID、REGION、SERVICE 等变量按目标环境替换。

部署平台
Google Cloud Run
镜像仓库
Artifact Registry
运行形态
Next.js standalone Docker image
默认服务名
searchkiller

部署前检查

  • 确认目标 GCP 项目已开通 billing，并且当前账号有 Cloud Run、Cloud Build、Artifact Registry、Secret Manager 和 IAM 管理权限。
  • 确认项目可以通过 npm run build 完成生产构建。
  • 确认 next.config.ts 使用 output: 'standalone'，Dockerfile 会运行 standalone server。
  • 准备运行时环境变量：GOOGLE_VERTEX_PROJECT、GOOGLE_VERTEX_LOCATION、EXA_API_KEY、ES_CLOUD_ID、ES_API_KEY。
  • 不要把敏感环境变量写进镜像，也不要提交到 git；生产环境统一通过 Secret Manager 注入。

部署命令

1. 设置部署变量

PROJECT_ID="your-gcp-project-id"
REGION="us-central1"
SERVICE="searchkiller"
REPO="searchkiller"
IMAGE="$REGION-docker.pkg.dev/$PROJECT_ID/$REPO/app:latest"

gcloud config set project "$PROJECT_ID"

2. 确认 gcloud 项目和账号

gcloud config get-value project
gcloud auth list --filter=status:ACTIVE --format='value(account)'

3. 启用依赖的 GCP 服务并创建镜像仓库

gcloud services enable \
  run.googleapis.com \
  cloudbuild.googleapis.com \
  artifactregistry.googleapis.com \
  secretmanager.googleapis.com \
  aiplatform.googleapis.com

gcloud artifacts repositories create "$REPO" \
  --repository-format=docker \
  --location="$REGION" \
  --description="Searchkiller Docker images"

4. Cloud Build 构建并推送镜像

gcloud builds submit --tag "$IMAGE" .

5. 授权 Cloud Run 调用 Vertex AI 并部署

PROJECT_NUMBER=$(gcloud projects describe "$PROJECT_ID" --format='value(projectNumber)')
RUN_SA="$PROJECT_NUMBER-compute@developer.gserviceaccount.com"

gcloud projects add-iam-policy-binding "$PROJECT_ID" \
  --member "serviceAccount:$RUN_SA" \
  --role roles/aiplatform.user \
  --quiet

gcloud run deploy "$SERVICE" \
  --image "$IMAGE" \
  --region "$REGION" \
  --platform managed \
  --port 8080 \
  --memory 1Gi \
  --cpu 2 \
  --timeout 300 \
  --allow-unauthenticated \
  --set-env-vars "GOOGLE_CLOUD_PROJECT=$PROJECT_ID,GOOGLE_CLOUD_REGION=$REGION,GOOGLE_VERTEX_PROJECT=$PROJECT_ID,GOOGLE_VERTEX_LOCATION=$REGION"

Secret 配置

Cloud Build 和 Cloud Run 不会自动读取开发机环境变量。部署时不要把敏感值打进镜像， 应将密钥写入 Secret Manager，再绑定到 Cloud Run 环境变量。

创建 Secret

printf '%s' "$EXA_API_KEY_VALUE" | gcloud secrets create exa-api-key --data-file=-
printf '%s' "$ES_CLOUD_ID_VALUE" | gcloud secrets create es-cloud-id --data-file=-
printf '%s' "$ES_API_KEY_VALUE" | gcloud secrets create es-api-key --data-file=-

更新已有 Secret

printf '%s' "$EXA_API_KEY_VALUE" | gcloud secrets versions add exa-api-key --data-file=-
printf '%s' "$ES_CLOUD_ID_VALUE" | gcloud secrets versions add es-cloud-id --data-file=-
printf '%s' "$ES_API_KEY_VALUE" | gcloud secrets versions add es-api-key --data-file=-

授予 Cloud Run 服务账号读取 Secret 的权限

PROJECT_NUMBER=$(gcloud projects describe "$PROJECT_ID" --format='value(projectNumber)')
RUN_SA="$PROJECT_NUMBER-compute@developer.gserviceaccount.com"

for s in exa-api-key es-cloud-id es-api-key; do
  gcloud secrets add-iam-policy-binding "$s" \
    --member "serviceAccount:$RUN_SA" \
    --role roles/secretmanager.secretAccessor \
    --quiet
done

绑定 Secret 到 Cloud Run

gcloud run services update "$SERVICE" \
  --region "$REGION" \
  --set-secrets EXA_API_KEY=exa-api-key:latest,ES_CLOUD_ID=es-cloud-id:latest,ES_API_KEY=es-api-key:latest

验证命令

首页健康检查

URL=$(gcloud run services describe "$SERVICE" --region="$REGION" --format='value(status.url)')
curl -s -o /dev/null -w '%{http_code} %{url_effective}\n' "$URL"

Vertex AI 规划接口

curl -s -X POST "$URL/api/plan" \
  -H 'Content-Type: application/json' \
  -d '{"keyword":"AI Agent 测试"}'

Exa / Elasticsearch 检索接口

curl -s -X POST "$URL/api/research/fetch" \
  -H 'Content-Type: application/json' \
  -d '{"keyword":"AI Agent 测试","subQueries":["AI Agent testing methodologies","AI Agent reliability evaluation","AI Agent benchmark frameworks"]}'

常见构建问题

  1. agents/evaluator/backend-judge.ts： 将 handoff.output.metrics 改为 handoff.metrics。 原因是 HandoffDocument 的 canonical 类型中 metrics 位于顶层，错误访问会导致 TypeScript 构建失败。
  2. lib/gemini.ts： 改用 createVertex 并显式传入 location 和 project。 原因是 Cloud Build 阶段没有开发机环境变量，默认 Vertex provider 会在 Next.js 收集页面数据时因缺少 GOOGLE_VERTEX_LOCATION 报错。
  3. Dockerfile： 在 npm run build 前增加 RUN mkdir -p public。 原因是项目当前没有 public/ 目录，但 production stage 会复制 /app/public。

注意事项

  • 开发环境变量不会自动进入 Cloud Build 或 Cloud Run；生产环境必须显式配置环境变量或 Secret。
  • Secret 更新后需要重新执行 gcloud run services update ... --set-secrets 或添加新 secret version 后重新部署 revision。
  • Cloud Run revision 的运行服务账号至少需要 roles/aiplatform.user 和对应 secret 的 roles/secretmanager.secretAccessor。
  • .github/workflows/deploy.yml 当前仍引用旧目录 Hackson/G-RapidAgent，如要启用 CI/CD，需要单独修正。
  • 线上服务当前允许匿名访问：--allow-unauthenticated。如果后续涉及私有数据，应加认证或限制 ingress。
  • Cloud Run 使用 --timeout 300，长报告生成如果超过 5 分钟仍会超时，需要改成后台任务或提高超时配置。
