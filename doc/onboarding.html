第一部分：程序架构

1.1 Searchkiller 是什么？

Searchkiller 是一个关键词驱动的流式调研智能体。用户输入任意探究性关键词（例如「Gemini 2.5 在医疗诊断中的多模态应用趋势」），系统会并行实时输出两份结果：

  • 左侧面板：结构化的 Markdown 分析报告
  • 右侧面板：交互式思维导图树

两个流同时推进 — 用户看到报告「逐字打出」的同时，脑图也在「生长」。

1.4 4 层架构

L0 harness/ 通用反馈循环框架 — IJudge, IReportGenerator 接口，零外部依赖
L1 agents/ 实现 harness 接口 — LLM-Judge, Planner, Evaluator
L2 app/ Next.js 页面 + API 路由 — 通过 lib/ 访问 harness
L3 lib/ 共享工具 + harness-adapter.ts（L0 与 L1 的唯一桥接）

Import 方向只能向下：app/ → lib/ → harness/ / agents/。app/ 不直接引用 agents/ 或 harness/。

第二部分：Harness Engineering（治理工程）

2.1 什么是 Harness Engineering？

仓库本身就是 Agent 的运行时。.agents/ 目录下的文件系统充当确定性状态机。每个 Agent 动作都会在磁盘上产生结构化记录。

三个核心原则：

  1. 仓库即 Trace 系统 — 每次动作写 JSON 到 .agents/
  2. 双 Markdown 隔离 — README.md（人）+ AGENT.md（机器）
  3. 接口解耦 — 通用框架 (harness/) 通过 IJudge / IReportGenerator 接口与业务逻辑 (agents/) 分离

2.2 反馈循环引擎

核心创新：解耦的迭代质量反馈循环，替代原来的单次评估模式。

Before（单次评估）

Planner → Evaluator → APPROVED / REJECTED

问题：无迭代、无反馈、无合约

After（迭代反馈循环）

Generator → Judge → 反馈 → 重生成 → 再评分 → ... → APPROVED

最多 10 轮迭代，4 维评分，Sprint 合约

四个 Harness 组件

组件文件职责
Handoff Protocolharness/handoff.tsAgent 间结构化上下文传递，文件存储
Sprint Contractharness/sprint-contract.ts协商式验收标准，4 维加权评分 + 硬阈值
LLM-as-Judgeagents/evaluator/llm-judge.tsGemini Flash 评分（thinkingBudget=0, ~4s）

新人快速上手

# 1. 克隆仓库
git clone git@github.com:ChatTrace-ai/Searchkiller.git
cd Searchkiller

# 2. 安装依赖
npm install

# 3. 配置环境变量
cp .env.example .env
# 填入: GOOGLE_CLOUD_PROJECT, GOOGLE_VERTEX_PROJECT, GOOGLE_VERTEX_LOCATION,
#       EXA_API_KEY, ES_CLOUD_ID, ES_API_KEY

# 4. 启动开发服务器
npm run dev

# 5. 运行测试
# 单元测试（无需 API Key）
npx tsx tests/handoff.unit.test.ts
npx tsx tests/sprint-contract.unit.test.ts

# 集成测试（需要 Vertex AI Key）
source .env
npx tsx tests/llm-judge.integration.test.ts

# 全管线测试（需要所有 Key）
npx tsx tests/full-pipeline.integration.test.ts

# 6. 验证仓库结构
bash scripts/verify-dual-md.sh
python scripts/lint-agent-md.py
