1. 概述

Searchkiller 使用 harness-engineering 模式：将仓库结构本身作为 agent 运行时。 核心创新是一个解耦的迭代质量反馈循环，将通用框架 (harness/) 与业务逻辑 (agents/) 分离。

Before（单次评估）

Planner → Evaluator → APPROVED / REJECTED

问题：无迭代、无反馈、无合约

After（迭代反馈循环）

Generator → Judge → 反馈 → 重生成 → 再评分 → ... → APPROVED

5-15 轮 HITL 迭代，4 维评分，Sprint 合约

2. 架构全景

2.1 4 层导入层级

L0

harness/ — 通用框架

types.ts · handoff.ts · sprint-contract.ts · feedback-loop.ts · index.ts
零外部依赖 — 任何项目可复用，定义 IJudge / IReportGenerator 接口
▼ implements
L1

agents/ — Searchkiller 业务逻辑

evaluator/ (确定性评估 + LLM-Judge) · planner/ (任务分解 + trace) · recycle.ts
handoff.ts / sprint-contract.ts → re-export from harness/
▼ adapts
L3

lib/ — 共享工具 + 桥接

harness-adapter.ts （唯一桥接点） · gemini.ts · exa.ts · elasticsearch.ts
将 agents/ 实现注入 harness/ 接口，对上层暴露统一 API
▼ uses
L2

app/ — 路由 & 页面

api/evaluate/ · api/research/ · research/ (前端 UI)
禁止 直接 import agents/ 或 harness/ — 只能通过 lib/harness-adapter

关键规则：app/ 永远不直接 import agents/ 或 harness/。所有调用通过 lib/harness-adapter.ts。

3. 工作流与迭代

3.1 HITL 反馈循环

sequenceDiagram
    actor User
    participant API as /api/evaluate
    participant Engine as FeedbackLoop
    participant Gen as IReportGenerator
    participant Judge as IJudge
    participant Store as .agents/

    User->>API: start_loop(keyword)
    API->>Engine: startFeedbackLoop()
    Engine->>Store: proposeContract → save
    Engine->>Gen: generate(round 1)
    Gen-->>Engine: report + sources
    Engine->>Judge: evaluate(handoff)
    Judge-->>Engine: scores + feedback
    Engine->>Store: save handoff + loop state
    Engine-->>API: {loop, round:1, score}
    API-->>User: round 1 结果 + 评分

    alt 用户满意
        User->>API: loop_approve(loopId)
        API->>Engine: approveLoop()
        Engine->>Store: 标记 golden
        Engine-->>User: ✅ APPROVED
    else 需要改进
        User->>API: loop_next(loopId, feedback?)
        API->>Engine: nextRound(feedback)
        Engine->>Gen: generate(round N, 含反馈)
        Gen-->>Engine: 改进后的 report
        Engine->>Judge: evaluate(handoff)
        Judge-->>Engine: 新评分
        Engine-->>User: round N 结果
    else 放弃
        User->>API: loop_cancel(loopId)
        Engine->>Store: 归档
        Engine-->>User: ❌ CANCELLED
    end

3.2 循环终止条件

✓ 正常终止

用户 loop_approve

✓ 自动批准

评分 ≥ autoApproveScore (8.5)

⚠ 到期

达到 maxRounds (10)

✗ 取消

用户 loop_cancel

3.5 典型迭代时间线

Round 1 — 初始生成

生成第一版报告 → 事实准确性 6.2 / 结构 5.8 / 深度 5.0 / 引用 4.5 → 加权 5.5

Round 2 — 用户反馈：增加引用

改进引用来源 → 事实 6.8 / 结构 6.5 / 深度 5.5 / 引用 6.0 → 加权 6.3

Round 3 — 用户反馈：深化分析

增加对比分析 → 事实 7.5 / 结构 7.0 / 深度 7.0 / 引用 6.8 → 加权 7.1 ✓

Round 4 — 自动优化

基于 Judge 反馈自动优化 → 事实 8.2 / 结构 8.0 / 深度 7.8 / 引用 7.5 → 加权 7.9

Round 5 — 用户批准

用户确认质量 → loop_approve() → 标记为 golden → APPROVED ✓

5. 核心组件

5.1 Handoff Protocol — harness/handoff.ts

Planner+Generator 和 Evaluator 之间的强类型合约。文件存储在 .agents/handoffs/。

接口说明
HandoffDocument核心文档：input (keyword, subQueries, userFeedback) + output (report, mindmap, sources) + metrics + evaluation + history
HandoffPhase状态机：created → evaluated → iterated → approved/rejected → archived
createHandoffDocument()创建新文档
createIterationHandoff(parentId, ...)创建迭代版本（通过 parentId 形成版本链）
resetContext()归档所有活跃 handoff，清空上下文

5.2 Sprint Contract — harness/sprint-contract.ts

Anthropic 启发的 Sprint 合约模式：在开始工作前，协商明确的验收标准。

1
proposeContract(keyword) — 系统提出默认合约
2
adjustContract(contract, adjustments) — 用户调整维度/阈值
3
activateContract(contract) — 锁定验收标准

5.3 LLM-as-Judge — agents/evaluator/llm-judge.ts

Gemini Flash 驱动的结构化质量评分，实现 IJudge 接口。

输出 Zod 验证的结构化 JSON：4 维分数 + 优点 + 不足 + 可执行改进建议。

5.4 Feedback Loop Engine — harness/feedback-loop.ts

整合 Handoff、Sprint Contract、IJudge 的完整迭代循环。通过依赖注入接收 Judge 和 Generator。

6. API 参考

所有操作通过 POST /api/evaluate，以 action 字段区分：

6.1 原有 Actions（单次评估）

Action输入说明
initializeinitialized_by, criteria, thresholdsHITL 初始化评估器
config—读取当前配置
evaluatekeyword, subQueries单次自主评估
evaluate_tracetraceId评估已有 trace
stats—系统统计

6.2 新增 Actions（反馈循环）

Action输入输出
start_loop{ keyword, subQueries?, sources?, maxRounds? }{ loop, latestRound, sprintComplete }
loop_next{ loopId, userFeedback? }{ loop, latestRound, sprintComplete }
loop_approve{ loopId }{ status, loop, summary }
loop_cancel{ loopId, reason? }{ status, loop }
loop_status{ loopId }{ loop, summary }
loop_list—{ loops, count }

6.3 curl 示例

# 启动反馈循环
curl -X POST http://localhost:3000/api/evaluate \
  -H "Content-Type: application/json" \
  -d '{
    "action": "start_loop",
    "keyword": "AI Agent 架构趋势",
    "subQueries": ["多Agent系统如何协作?", "Agent记忆机制有哪些?"],
    "maxRounds": 10
  }'

# 查看循环状态
curl -X POST http://localhost:3000/api/evaluate \
  -H "Content-Type: application/json" \
  -d '{"action": "loop_status", "loopId": "loop-xxx"}'

# 下一轮迭代（带用户反馈）
curl -X POST http://localhost:3000/api/evaluate \
  -H "Content-Type: application/json" \
  -d '{
    "action": "loop_next",
    "loopId": "loop-xxx",
    "userFeedback": "请增加对Agent安全性的分析"
  }'

# 批准当前最佳版本
curl -X POST http://localhost:3000/api/evaluate \
  -H "Content-Type: application/json" \
  -d '{"action": "loop_approve", "loopId": "loop-xxx"}'
