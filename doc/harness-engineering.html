1. 概述

Searchkiller 使用 harness-engineering 模式：将仓库结构本身作为 agent 运行时。 核心创新是一个解耦的迭代质量反馈循环，将通用框架 (harness/) 与业务逻辑 (agents/) 分离。

Before（单次评估）

Planner → Evaluator → APPROVED / REJECTED

问题：无迭代、无反馈、无合约

After（迭代反馈循环）

Generator → Judge → 反馈 → 重生成 → 再评分 → ... → APPROVED

5-15 轮 HITL 迭代，4 维评分，Sprint 合约

2. 架构：4 层导入层级

L0 harness/ ← 通用框架（零外部依赖，任何项目可复用） types.ts IJudge, IReportGenerator 接口 handoff.ts 结构化 Handoff 协议 sprint-contract.ts Sprint 合约 + 4 维评分 feedback-loop.ts 迭代反馈循环引擎 index.ts Barrel export ↑ implements L1 agents/ ← Searchkiller 业务逻辑 evaluator/ 确定性评估 + LLM-as-Judge (Gemini) planner/ 任务分解 + trace 追踪 recycle.ts 回收模式引擎 handoff.ts re-export → harness/handoff sprint-contract.ts re-export → harness/sprint-contract ↑ adapts L3 lib/ ← 共享工具 harness-adapter.ts 唯一桥接点：harness + agents gemini.ts Vertex AI 模型 exa.ts / elasticsearch.ts 搜索引擎 ↑ uses L2 app/ ← 路由和页面 api/evaluate/ 通过 lib/harness-adapter 调用 api/research/ 通过 lib/ 直接调用 research/ 前端 UI

关键规则：app/ 永远不直接 import agents/ 或 harness/。所有调用通过 lib/harness-adapter.ts。

3. 核心组件

3.1 Handoff Protocol — harness/handoff.ts

Planner+Generator 和 Evaluator 之间的强类型合约。文件存储在 .agents/handoffs/。

接口说明
HandoffDocument核心文档：input (keyword, subQueries, userFeedback) + output (report, mindmap, sources) + metrics + evaluation + history
HandoffPhase状态机：created → evaluated → iterated → approved/rejected → archived
createHandoffDocument()创建新文档
createIterationHandoff(parentId, ...)创建迭代版本（通过 parentId 形成版本链）
resetContext()归档所有活跃 handoff，清空上下文

3.2 Sprint Contract — harness/sprint-contract.ts

Anthropic 启发的 Sprint 合约模式：在开始工作前，协商明确的验收标准。

1
proposeContract(keyword) — 系统提出默认合约
2
adjustContract(contract, adjustments) — 用户调整维度/阈值
3
activateContract(contract) — 锁定验收标准

3.3 LLM-as-Judge — agents/evaluator/llm-judge.ts

Gemini Flash 驱动的结构化质量评分，实现 IJudge 接口。

输出 Zod 验证的结构化 JSON：4 维分数 + 优点 + 不足 + 可执行改进建议。

3.4 Feedback Loop Engine — harness/feedback-loop.ts

整合 Handoff、Sprint Contract、IJudge 的完整迭代循环。通过依赖注入接收 Judge 和 Generator。

4. API 参考

所有操作通过 POST /api/evaluate，以 action 字段区分：

原有 Actions（单次评估）

Action输入说明
initializeinitialized_by, criteria, thresholdsHITL 初始化评估器
config—读取当前配置
evaluatekeyword, subQueries单次自主评估
evaluate_tracetraceId评估已有 trace
stats—系统统计

新增 Actions（反馈循环）

Action输入输出
start_loop{ keyword, subQueries?, sources?, maxRounds? }{ loop, latestRound, sprintComplete }
loop_next{ loopId, userFeedback? }{ loop, latestRound, sprintComplete }
loop_approve{ loopId }{ status, loop, summary }
loop_cancel{ loopId, reason? }{ status, loop }
loop_status{ loopId }{ loop, summary }
loop_list—{ loops, count }

curl 示例

# 启动反馈循环
curl -X POST http://localhost:3000/api/evaluate \
  -H "Content-Type: application/json" \
  -d '{
    "action": "start_loop",
    "keyword": "AI Agent 架构趋势",
    "subQueries": ["多Agent系统如何协作?", "Agent记忆机制有哪些?"],
    "maxRounds": 10
  }'

# 查看循环状态
curl -X POST http://localhost:3000/api/evaluate \
  -H "Content-Type: application/json" \
  -d '{"action": "loop_status", "loopId": "loop-xxx"}'

# 下一轮迭代（带用户反馈）
curl -X POST http://localhost:3000/api/evaluate \
  -H "Content-Type: application/json" \
  -d '{
    "action": "loop_next",
    "loopId": "loop-xxx",
    "userFeedback": "请增加对Agent安全性的分析"
  }'

# 批准当前最佳版本
curl -X POST http://localhost:3000/api/evaluate \
  -H "Content-Type: application/json" \
  -d '{"action": "loop_approve", "loopId": "loop-xxx"}'

5. HITL 工作流

用户调用 start_loop(keyword) │ ▼ 后端生成第1版报告 → LLM-as-Judge 评分 → 返回 {round:1, score, feedback} │ ▼ 用户查看评分和反馈 ├── 满意 → loop_approve(loopId) → 标记 golden，结束 ├── 不满意 → loop_next(loopId, userFeedback?) → 生成第2版... └── 放弃 → loop_cancel(loopId) → 归档 循环终止条件： • 用户 approve • 评分达到 autoApproveScore（默认 8.5）→ 自动 approve • 达到 maxRounds 上限（默认 10）→ 状态变为 expired

6. 评分标准

4 维加权评分

维度权重硬阈值目标说明
事实准确性
factual_accuracy		30%5.07.5信息与源数据一致，无捏造
结构完整性
structural_completeness		25%5.07.0必要章节完整，逻辑连贯
分析深度
analysis_depth		25%4.07.0超越表面描述，有洞察力
引用质量
citation_quality		20%4.06.5引用充分、准确、来源可靠

全局阈值

参数默认值说明
minWeightedScore7.0加权总分最低要求
autoApproveScore8.5达到此分数自动批准
maxRounds10最大迭代轮数
timeoutMinutes30超时自动归档

合约满足条件：所有硬阈值通过 且 加权总分 ≥ minWeightedScore。

7. 文件存储协议

.agents/
├── evaluator-config.json     # HITL 初始化的评估配置
├── traces/                   # 执行追踪记录
│   └── {traceId}.json
├── golden/                   # 通过审批的高质量结果
│   └── {traceId}.json
├── failures/                 # 未通过的结果
│   └── {traceId}.json
├── handoffs/                 # 结构化 Handoff 文档
│   ├── {handoffId}.json
│   └── _archive/             # 归档的旧 Handoff
├── contracts/                # Sprint 合约
│   └── {sprintId}.json
└── loops/                    # 反馈循环状态
    └── {loopId}.json
